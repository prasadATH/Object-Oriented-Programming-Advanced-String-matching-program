4.1   Reading  a Software  Project

The  application must  take   an   optional   command-line  parameter, which  must  be the  name   of   a   directory.  If no  directory  name is  given,   then   the   current  directory “.”  must  be used  instead. ̢̡̨̨̧̠̣̥

The   application   must   read  all files  within  the directory,  recursively   (see   Reading   Directory  Trees), and  store  their names  and   contents   in   memory.  It will  also  need to  store   the   names   of  all directories  in  the tree.  We   assume   that   the  directory tree  contains  the source  files   for   a   software  project (though  it  could actually  contain   any   text   files). ̢̡̨̨̧̠̣̥

4.2  Menu

After having  loaded  the software  project’s   source   files,   the  application must  present  a simple  text-based   menu   system   with  four options,  as  follows: ̢̡̨̨̧̠̣̥

  1. “Set Criteria”.  When   selected,   the   user  will be  asked  to enter  a   series   of   criteria.  (See Entering  Search  Criteria.) ̢̡̨̨̧̠̣̥

    ̢̡̨̨̧̠̣̥

  2. “Set Output  Format”.   When   selected,   the  user will  be  asked to  choose   between   available   output  formats. (See  Choosing  the Output  Format.) ̢̡̨̨̧̠̣̥

  3. “Report”.   When   selected,   the  application will  recurse  through the  loaded   software   project,   searching  for lines  that  match the  given   criteria   and   produce  output as  per  the output  format.   (See   Reporting.) ̢̡̨̨̧̠̣̥

  4. “Quit”.   Self-explanatory.  The menu  system  should not  exit   until   the   user  selects Quit. ̢̡̨̨̧̠̣̥

    ̢̡̨̨̧̠̣̥

4.3  Entering  Search Criteria

The  application   maintains   several   search  criteria. These  criteria  are to  be   entered   by   the  user after  selecting  the “Set  Criteria”   option.   The   application  should accept  several  lines of  console   user   input   at  once, ending  in  a blank  line.   Each   non-blank   line  should be  entered  in the  following   format:   “i”   or  “e”, space,  “t”  or “r”,  space,   remaining   text.   The  remaining text  must  contain at  least   one   character,   and  can contain  any  characters except  new-lines   (“\n”)   or   carriage  returns (“\r”). ̢̡̨̨̧̠̣̥

The  “i”  and “e”  mean   “include”   and   “exclude”,  respectively, while  “t”  and “r”  mean   “text”   and   “regular  expression” (or  “regex”),  respectively (see  Regular   Expressions).   The   remaining  text is  either  plain text,  or   a   properly-formatted   regular  expression, as  appropriate.  Thus, each  criterion   can   be   a  text inclusion,  regex  inclusion, text  exclusion   or   regex   exclusion,  and there  can  be any  number   of   each. ̢̡̨̨̧̠̣̥

For   example,  the user  might  enter the  following   set   of   criteria:
̢̡̨̨̧̠̣̥

i t Hello
e t Au revoir!
i r ^(berry)*
e r ba(na)+$
i t world
e r [aeiou]

(This  which would  end  in a  blank   line,   though   the  blank line  itself  doesn’t count.)

There  is   a   single   default  criterion: “i r .*”  (which  causes all  lines   to   be   included,  as we’ll  see  below). Each  time   the   user   selects  “Set Criteria”,  the  previous set  of   criteria   (initially   set  to the  default)  is entirely  replaced   with   the   newly-entered  criteria. ̢̡̨̨̧̠̣̥

4.4 Choosing  the  Output Format

When  the   user   selects   “Set  Output Format”,  they  must then  choose   between   two   options: ̢̡̨̨̧̠̣̥

  • “Count  of matching  lines  per file”  (abbreviated   as   “count”   from  here on); ̢̡̨̨̧̠̣̥
  • “Show  actual  matching lines”  (abbreviated   as   “show”   from  here on). ̢̡̨̨̧̠̣̥

Initially,  the  output format  is   set   to   “count”,  and will  be  overwritten each  time   the   user   makes  a choice  under  “Set Output  Format”. ̢̡̨̨̧̠̣̥

4.5   Reporting

When   the   user  selects “Report”,  the  application will  search   through   each   file  in the  loaded  software project,  line-by-line.   For   each   line,  it checks  the  criteria to  determine   whether   to   “include”  the line.  All  “included” lines  are   acted   upon   according  to the  output  format. ̢̡̨̨̧̠̣̥

A text  criterion   “matches”   a   line  if any  part  of the  line   (any   substring)   is  exactly equal  to  the given  text.   A   regex   criterion  “matches” a  line  if any  part   of   the   line  matches the  regular  expression (as  per   the   Matcher.find()   method,  explained in  Regular Expressions  below). ̢̡̨̨̧̠̣̥

For each  line   in   a   file,  the application  must  first test  all   the   exclusion   criteria,  if any.  If  any match,  then   the   line   is  ignored. Otherwise,  it  must test  all   the   inclusion   criteria,  if any.  If  any match,  the   line   is   “included”.  Otherwise, if  no  criteria at  all   match,   the   line  is ignored. ̢̡̨̨̧̠̣̥

If  the  output format  is   “count”,   then,   for  each file,  the  application outputs  the   filename   and   the  number of  included  lines. It  also   outputs   the   name  and number  of  matching lines  for   each   directory   (within  the project).  For  a directory,  the   count   of   matching  lines is  the  sum of  the   matching   lines   of  all files  and  subdirectories inside  that   directory. ̢̡̨̨̧̠̣̥

4.6.1  Command-Line   Parameters

Command-line   parameters   are  available to  a  Java program  via   the   argument   to  main(); i.e.: ̢̡̨̨̧̠̣̥

public static void main(String[] args)
{

    {

    }

    {

}

̢̡̨̨̧̠̣̥

$ java MainClass some/directory

(Swap  “/” with  “\”   on   Windows.) ̢̡̨̨̧̠̣̥

4.6.2   Reading  Directory Trees

There  are  various ways  to   find   the   contents  of a  directory,  or directory  tree,   in   Java.   You  may use  any  reliable approach,  but   you   may   find  File.listFiles() the  simplest  one. ̢̡̨̨̧̠̣̥

Given a  File   object   (which   can  represent a  file  or a  directory),   we   can   call  listFiles() to  get  an array  of   the   contents   of  the directory  (or  null if  it’s   not   a   directory): ̢̡̨̨̧̠̣̥

import java.io.File;
...
File dir = new File("some/directory");
File[] dirContents = dir.listFiles();

Doing  this recursively  will  give you  the   contents   of   the  entire tree. ̢̡̨̨̧̠̣̥

Alternatively: ̢̡̨̨̧̠̣̥

  • You  could  call File.list()  to   get   an   array  of Strings,  rather  than File  objects,   but   this   is  probably less  useful. ̢̡̨̨̧̠̣̥
  • You  can use  Files.walkFileTree()   (note:   Files,   not  File), which  takes  a FileVisitor  object   that   you   must  create, and  calls  it for  every   file   in   the  directory tree.  (Incidentally,  this uses  the   “Visitor   Pattern”,   which  isn’t really  covered  in this  unit,   but   which   you  may find  interesting.) ̢̡̨̨̧̠̣̥
  • You  can use  Files.walk(),   which   returns   a  Stream object  (a  bit like  a   fancy   iterator)   that  iterates over  every  file in  the   directory   tree. ̢̡̨̨̧̠̣̥

4.6.3   Regular  Expressions

Java has  a  Pattern class  for   performing   regular   expression  (“regex”) matching.  (This  class has  nothing   whatsoever   to   do  with the  design  patterns we’ve  covered!   It   just   happens  to be  useful  in this  context.) ̢̡̨̨̧̠̣̥

It   isn’t   necessary   to  understand regular  expressions  in detail,  for   the   purpose   of  this assignment,  because  the user  is   responsible   for   creating  them. However,  you  will need  to   understand   the   general  idea, and  be  able to  test   them.   A   regular  expression is  a  string containing  plain   text   as   well  as certain  special  symbols, which  let   it   “match”   other  strings: ̢̡̨̨̧̠̣̥

  • “.” means  any  single character; ̢̡̨̨̧̠̣̥
  • “*”  means   zero-or-more   of   the  preceding item  (so  that “.*”  means   any   sequence   of  characters). ̢̡̨̨̧̠̣̥
  • “+” means  one-or-more  of the  preceding   item; ̢̡̨̨̧̠̣̥
  • “?”   means   zero-or-one  of the  preceding  item; ̢̡̨̨̧̠̣̥
  • “(...)” groups  a   set   of   characters  (so that  “(abc)*”  means zero-or-more  occurrences   of   “abc”); ̢̡̨̨̧̠̣̥
  • “[...]”   means  any single  character  from those  inside   the   brackets; ̢̡̨̨̧̠̣̥
  • “^”   means  the start  of  the text  (so   that   “^abc”   means  “abc”, but  only  at the  beginning   of   the   string); ̢̡̨̨̧̠̣̥
  • “$​”  means the  end  of the  text   (so   that   “abc$​”  means “abc”,  but  only at  the   end   of   the  string); ̢̡̨̨̧̠̣̥
  • “\” is  the  escape character,  which   you   can   put  before a  special  character like  “.”,   “*”,   “\”   itself,  etc. to  remove  its special  meaning   (so   that   “\*”  refers just  to  the literal  character   “*”,   not   to  zero-or-more of  anything); ̢̡̨̨̧̠̣̥
  • others  too, which  you   can   look   up  for yourself. ̢̡̨̨̧̠̣̥

You  can  use the  Pattern   class   as   follows,  given both  the  regular expression  string   and   the   string  you want  to  match it  against: ̢̡̨̨̧̠̣̥

  1. Obtain   a   Pattern   object  by calling  Pattern.compile(),  and passing  the   regular   expression   string  as a  parameter.   ̢̡̨̨̧̠̣̥

    ̢̡̨̨̧̠̣̥

  2. Obtain a  Matcher   object   by   calling  matcher() on  the  Pattern object,  passing   the   string   you  want to  match  the regular  expression   against   as   a  parameter. ̢̡̨̨̧̠̣̥

    ̢̡̨̨̧̠̣̥

  3. Call the  Matcher  object’s find() method.  It   will   return   true  if the  regular  expression matches  some   part   of   the  text, and  false  otherwise. ̢̡̨̨̧̠̣̥

    ̢̡̨̨̧̠̣̥

For illustration:
̢̡̨̨̧̠̣̥

import java.util.regex.Pattern;
import java.util.regex.Matcher;
...
String regex = "(abc)*";
String text  = "Some text abcabcabc some text";

Pattern p = Pattern.compile(regex);
Matcher m = p.matcher(text);
if(m.find())
{
    System.out.println("Match found");
}

Important ̢̡̨̨̧̠̣̥

There  are   ways   for   regular  expressions to  be  invalid, such  as   mismatching   ()   brackets. ̢̡̨̨̧̠̣̥
